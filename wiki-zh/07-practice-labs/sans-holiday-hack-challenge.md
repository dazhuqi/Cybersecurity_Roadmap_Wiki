SANS Holiday Hack Challenge 是季节性挑战系列。它有技术任务、故事、目标、hints 和最终报告，所以感觉更像一个小调查，而不是单个孤立谜题。

我应该把它当慢练习：做笔记、收集证据、解释每个 objective、写干净的最终报告。这很适合练沟通，不只是技术求解。

## 为什么有用

这个挑战会混合：

- web tasks
- forensics
- cloud 或 identity ideas
- command-line puzzles
- investigation notes
- report writing
- hints 和 narrative context

报告写作很重要。真实工作里，默默解出来还不够，我需要解释发生了什么和为什么重要。

## 我想怎么用

- 动手前阅读 event rules 和 scope。
- 每个 objective 单独追踪。
- 只有截图能作为证据时才保存。
- 写下 hints、失败想法和最终推理。
- 练习像 lab report 一样写最终总结。
- writeups 按年份标清楚，因为内容会变。

## Objective 笔记格式

```text
Year:
Objective:
Status:
Location or challenge area:
Prompt:
Evidence collected:
Tools used:
Hint used:
What worked:
What failed:
Final answer or result:
Lesson:
```

## 证据习惯

有用证据：

- challenge prompt
- 相关 screenshot
- command output summary
- 重要 file hash 或 name
- packet 或 log clue
- final reasoning

没那么有用：

- 没解释的大截图
- 没笔记的 raw terminal dumps
- 从别人那里复制的 spoilers
- 暴露私人账号细节的截图

## 最终报告结构

干净报告可以包括：

1. Short overview。
2. Objective list。
3. 每个 objective 的 method。
4. 每个 objective 的 evidence。
5. Final answers 或 outcomes。
6. Lessons learned。
7. Appendix with commands or screenshots。

我应该用普通语言写。未来的我应该能读报告理解路径，而不是重开每个 tab。

## 使用 Hints

Hints 是学习的一部分，但我要诚实记录。

使用 hint 时写下：

- hint 前尝试了什么
- hint 澄清了什么
- 缺了哪个概念
- 下次能否解决相似任务

这样 hints 才有学习价值，而不是隐形跳步。

## 安全和范围

- 留在官方 challenge environment 内。
- 不攻击无关 SANS 系统或真实外部服务。
- 不发布敏感账号信息、tokens 或 private challenge state。
- writeups 按年份标清楚，因为 challenge content 会变。
- 遵守活动发布和参与规则。
